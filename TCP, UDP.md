### TCP

데이터의 신뢰성과 순서를 중요시할 때 사용하는 전송 계층 프로토콜입니다.

### UDP

신뢰성보다 속도와 효율성을 중요시할 때 사용하는 전송 계층 프로토콜

### 포트 번호

port number은 한 컴퓨터 안에서 어떤 프로그램에게 데이터를 전달할지 구분하는 번호입니다.

- IP 주소 = 아파트 주소
- 포트 번호 = 세부

| **포트 번호** | **프로토콜** | **설명** |
| --- | --- | --- |
| 20, 21 | FTP | 파일 전송 |
| 22 | SSH | 원격 접속 |
| 23 | Telnet | 원격 접속 |
| 25 | SMTP | 이메일 전송 |
| 53 | DNS | 도메인 이름 조회 |
| 80 | HTTP | 웹 페이지 |
| 443 | HTTPS | 암호화된 웹 페이지 |

포트 번호 범위

- 0 ~ 1023 : Well-Know Port (잘 알려진 포트)
    - HTTP(80), HTTPS(443) 등이 사용
- 1024 ~ 49151 : Registered Port
    - 특정 프로그램들이 사용
- 49152 ~ 65535 : Dynamic / Private Port
    - 클라이언트가 임시로 사용

ex)

브라우저로 구글 접속 시

1. 브라우저가 임시 포트(ex. 52341)를 사용
2. 구글 서버의 443번 포트(HTTPS)로 접속
3. 서버가 응답을 보내면 운영체제가 52341번 포트를 보고 브라우저에 전달

### TCP 특징

- 연결 지향 방식 (3-Way Handshake 사용)
- 데이터가 순서대로 도착하는지 확인
- 데이터 손실 시 재전송
- 오류 검사 및 흐름 제어 지원
- 속도는 UDP보다 상대적으로 느림

### UDP 특징

- 비연결형 방식
- 데이터 도착 여부 확인 안 함
- 재전송 없음
- 순서 보장 안 함
- TCP보다 빠름

### 3-Way Handshake

3개의 패킷을 주고 받으며 사전 연결 설정을 수립하는 3단계 과정입니다.

1단계 : 클라이언트 → 서버 (SYN)

클라이언트는 TCP 헤더의 SYN 플래그 비트를 1로 설정합니다. 무작위의 sequence 번호 (ed.A)를 세그먼트에 넣어서 서버에게 보냅니다.

2단계 : 서버 → 클라이언트 (SYN - ACK)

서버는 SYN 플래그와 ACK 플래그의 비트를 1로 설정합니다. Acknowledgement Number 필드에는 수신된 sequence 번호보다 하나 더 큰 값 (ex. Z)를 담고, 해당 세그먼트를 클라이언트에게 전송합니다.

3단계 : 클라이언트 → 서버 (ACK)

마지막으로 클라이언트는 ACK를 서버로 다시 보냅니다. Acknowlegment number는 서버가 보낸 sequence number 필드 값 (Z) 에 1을 더한 Z + 1로 설정됩니다. sequence number 필드에 서버가 보낸 Acknowledgement number 필드 값이었던 A + 1로 설정합니다. 이 때, 전송할 데이터가 있으면 이 단계에서 데이터를 전송하고, 서버는 클라이언트의 ACK 를 통해 현재 클라이언트가 alive임을 확인합니다.

### 4-Way Handshake

4개의 패킷을 주고 받으며 TCP 연결을 종료하는 과정입니다.

1단계 : 클라이언트 → 서버 (FIN)

클라이언트는 서버에게 연결을 종료하겠다는 FIN 플래그를 1로 설정한 세그먼트를 보냅니다.

2단계 : 서버 → 클라이언트 (ACK)

서버는 클라이언트에게 FIN에 대한 ACK를 보냅니다. ACK를 받은 클라이언트는 포트 상태를 FIN_WAIT_2로 바꾸고, 서버의 FIN 패킷을 기다립니다.

3단계 : 서버 → 클라이언트 (FIN)

서버가 클라이언트에게 연결을 종료하겠다는 FIN 플래그 비트를 1로 설정한 세그먼트를 보냅니다.

4단계 : 클라이언트 → 서버 (ACK)

클라이언트는 서버의 종료 요청에 응답하여 마지막 ACK를 보내고, 마지막 ACK를 수신한 서버는 포트를 닫습니다.
